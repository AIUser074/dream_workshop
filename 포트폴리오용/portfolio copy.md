## 1. 프로젝트 개요

`꿈의 그림 공방`은 사용자가 그림을 그려 손님의 의뢰를 해결하는 힐링 드로잉 게임입니다.

**이 프로젝트는 단순한 이미지 인식 데모가 아니라, 유저가 그린 엉뚱한 그림에도 AI가 맥락에 맞게 반응하는
게임 경험을 만드는 것을 목표로 시작되었습니다.**

겉으로 보면 감성적인 캐주얼 게임이지만, 실제로는 다음 세 가지를 동시에 풀어야 하는 고난도의 문제였습니다.

1. **그림을 보고 맥락을 이해하는 AI**
2. **게임의 보상 체계를 무너뜨리지 않는 일관된 판정**
3. **1인 개발이 감당 가능한 지속 가능한 비용 구조**

저는 이 프로젝트를 통해 기획, 프론트엔드, AI 프롬프트 설계, 서버리스 API, 런칭, 심의 대응, 스토어 등록, 수익화까지 직접 End-to-End로 구축했습니다.

## 2. 직면한 문제: "LLM 만능주의"의 한계

초기 프로토타입은 NPC 성격, 감정, 피드백, 점수까지 **모두 한 번의 LLM 호출로 생성하는 구조**였습니다.

같은 그림을 넣어도 호출마다 점수가 달라졌고, 유저에게는 보상이 "실력"이 아니라 "운"처럼 느껴지는 문제가 발생했습니다. AI가 똑똑한가보다 먼저, **게임 규칙이 믿을 만한가**가 중요했기 때문입니다.

### ⚠️ 초기 방식의 실제 프롬프트

```
(System) 당신은 꿈의 그림 공방의 '토마스'입니다. 깐깐하지만 관대한 성격을 가지고 있으며, '하게체'를 씁니다.
사용자가 제출한 그림을 보고 다음을 수행하세요:

1. 그림이 '파도를 가르는 돌고래'가 맞는지 분석하세요.
2. 당신의 호불호(주제와 맞는 그림을 좋아함)에 따라 감정을 결정하세요.
3. 그 감정을 바탕으로, 하게체로 3문장 이상의 피드백을 작성하세요.
4. 0점부터 100점 사이의 점수를 JSON으로 매겨주세요.
```

### 💸 초기 방식의 실제 토큰 사용량 (1회 호출 기준)

| 항목 | 수치 |
| --- | --- |
| Input tokens | `403` |
| Output tokens | `839` |
| **Total tokens** | **`1,242`** |

핵심 문제는 **점수의 비일관성**이었습니다.

점수 산정을 모델에게 통째로 맡기면, 게임 보상 체계와 유저 경험(UX) 자체가 흔들릴 수밖에 없었습니다.

## 3. 아키텍처 개편: AI를 통제 가능한 부품으로 설계하다

![image.png](attachment:495f0cd5-99db-40b5-a6cb-2c865112312b:image.png)

이 프로젝트의 해결책은 더 똑똑한 모델을 쓰는 것이 아니라, **LLM의 역할을 과감하게 줄이는 것**이었습니다.

> **"LLM은 그림을 이해하는 데까지만 쓰고, 점수와 보상은 반드시 클라이언트가 쥐어야 한다."**
> 

이 철학을 바탕으로, 의뢰의 목적에 맞게 두 가지 하이브리드 아키텍처를 설계했습니다.

### **3-1. `draw` 모드: 실력 평가 (비용 최적화 단일 호출)**

이 모드는 "주제를 잘 그렸는가"를 평가합니다. 감성 코멘트보다 **채점 기준의 일관성**이 중요하므로, 단 한 번의 호출로 `true/false`, `0/1/2` 값만 담긴 JSON을 받습니다.

또한, Vision API로 이미지를 전송하기 전, Canvas 원본을 그대로 보내지 않고 다음 전처리를 수행했습니다.

- 불필요한 빈 여백 자르기 (Crop)
- 최대 해상도 축소 (Resize)
- JPEG 80% 압축 (Base64 전송)

**[이미지 전처리 코드 발췌]**

```jsx
// draw.js 내 전처리 로직
const tempCanvas = document.createElement('canvas');
const tempCtx = tempCanvas.getContext('2d');
const ratio = Math.min(MAX_DIMENSION / cropW, MAX_DIMENSION / finalCropH, 1);

tempCanvas.width = cropW * ratio;
tempCanvas.height = finalCropH * ratio;

// 원본에서 크롭 영역만 추출하여 리사이징 후 다시 그리기
tempCtx.drawImage(this.canvas, cropX, cropY, cropW, finalCropH, 0, 0, tempCanvas.width, tempCanvas.height);

// 가벼운 JPEG로 압축하여 Base64 변환
const dataUrl = tempCanvas.toDataURL('image/jpeg', 0.8);
```

**[프롬프트 구조 개선]**

```json
[TASKS]
1. 그림에 '토마토'가 포함되어 있는가? (true/false)
2. 그림 속 '토마토'가 주로 '빨간색'인가? (true/false)
3. 그림 속 '토마토'의 모양이 '둥근' 편인가? (true/false)
4. 완성도 (0, 1, 2)

[ANSWER FORMAT]
{ "isTomato": ..., "isRed": ..., "isRound": ..., "detailLevel": ... }
```

이렇게 받아온 JSON 객관식 지표를 바탕으로, **점수는 100% 클라이언트(게임 엔진)가 결정**합니다. 모델이 환각을 일으켜 갑자기 100점을 주는 상황을 원천 차단했습니다.

**[결정론적 점수 계산 코드 발췌 (실제 구현부)]**

```jsx
// main.js 내 rubrics 점수 산정 로직
function processEvaluation(requestDef, llmResults) {
  let finalScore = 0;

  for (const rule of requestDef.rubrics) {
    const res = llmResults[rule.id]; // LLM이 내린 true/false 혹은 0/1/2

    let factor = 0;
    if (res === true) factor = 1;
    else if (typeof res === 'number') factor = Math.max(0, Math.min(2, res)) / 2;

    // 클라이언트에 하드코딩된 가중치(weight) 기반으로 점수 합산
    finalScore += (rule.weight || 0) * factor;
  }

  return { score: finalScore };
}
```

이 방식을 통해 초기 `1,242` 토큰에서 `689` 토큰으로 비용을 약 **44.5% 절감**함과 동시에, 같은 그림에 대해 언제나 같은 점수를 보장하는 완벽한 게임 룰을 확립했습니다.

### **3-2. `solve` 모드: 문제 해결 아이디어 평가 (2-Step 호출)**

`solve` 모드는 그림 실력보다 문제를 해결하는 "아이디어"가 중요합니다. 한 번의 호출로 생성하면 점수 일관성과 스토리 분기가 모두 불안정해졌습니다. 그래서 이 모드는 두 단계로 나눴습니다.

**Step 1. ANALYZE (객관적 분석)**

```json
[분석 항목]
is_drawing: 이 입력이 명확한 '그림'인가?
is_meal: 먹을 수 있는 음식인가?
is_dinner: 일반적인 저녁 메인 메뉴인가?
```

**Step 2. CREATE (감성적 반응 생성)**

클라이언트가 `ANALYZE` 결과로 스토리 분기를 먼저 결정하고, 그 상황을 LLM에 넘겨 **단 한 줄의 NPC 코멘트**만 텍스트로 생성합니다.

```json
[SITUATION] 유저는 '맛있는 저녁 식사'를 그려주어 '문제가 해결된 상황'입니다.
[TASK] 위 상황과 '토마스'의 페르소나에 맞춰 한 줄 코멘트만 { "comment": "..." }로 생성하세요.
```

초기 구조에서는 응답 시간이 평균 4~5초 수준이었습니다. 하지만 이 방식을 통해 `draw` 기준 **2초대**, `solve` 역시 **3초 내외**로 응답을 단축시켜, 모바일 게임에 적합한 실시간(UX) 경험을 달성했습니다. 전체 `solve` 비용도 약 **30.7% 절감**(`861` 토큰)했습니다.

### **3-3. 공통 방어 로직: Early Exit**

실서비스에서는 빈 캔버스, 텍스트 입력, 의미 없는 낙서 등 다양한 입력이 발생합니다. 이 무의미한 입력들까지 굳이 두 번째 반응 생성 단계로 넘기면 비용과 지연만 낭비됩니다.

그래서 공통 분석 항목으로 `is_drawing`을 반드시 먼저 검사하고, 이 값이 `false`면 즉시 게임을 실패로 간주하고 생성 호출을 생략하는 방어 코드를 작성했습니다.

**[Early Exit 방어 로직 발췌]**

```jsx
// draw.js 내 선판정 로직
const analyzeRes = await window.LLMService.evaluateDrawing({ dataUrl, meta: analyzeMeta });
const analysis = analyzeRes.analysis || {};

if (normalizeBool(analysis.is_drawing) === false) {
  // 그림이 아니면 2차 호출을 생략하고 즉각 최하점 반응 처리 (Early Exit)
  return this._handleSubmissionResult({
    score: 12,
    comment: '이건 그림이 아니잖아요....',
    solveOutcome: 'reaction_terrible'
  });
}

// 이후 CREATE 모드 등 후속 로직 진행...
```

이 분기 하나로 `solve` 전체 경로 `861 tokens` 중 2차 생성 `254 tokens`를 바로 잘라낼 수 있습니다. 

 이는 전체 `solve` 경로 대비 **약 29.5% 추가 절감**입니다.

## 4. 서버리스 보안과 운영 인프라 설계

이 프로젝트는 프론트엔드에서 직접 LLM을 호출하지 않습니다. 클라이언트는 오직 `/api/evaluate` 엔드포인트만 호출하고, 실제 모델 선택과 API 키 관리는 Vercel 서버리스 함수가 담당합니다.

![image.png](attachment:1d73b738-7ba6-4b56-b880-8bf8389c2881:image.png)

**[운영 안정성을 위해 의도적으로 설계한 인프라]**

- **보안 격리:** API Key는 클라이언트에 두지 않고 Vercel 환경변수(ENV)에서만 읽어 탈취를 원천 차단.
- **제어권 분리:** 클라이언트는 모델 순서나 키를 모르고, 오직 서버가 Provider Order를 결정함.
- **크로스 도메인 대응:** CORS 헤더와 `OPTIONS` Preflight 응답을 서버에서 처리해 외부 WebView 환경(토스 앱)에서도 호출을 안정화함.
- **고가용성 설계 (Fallback):** Gemini / OpenAI를 순차 폴백하도록 설계해 단일 벤더 API 장애 시 시스템 복원력을 확보함.
- **예외 처리 강화:** JSON 형식을 강제하더라도 LLM이 간헐적으로 Markdown 코드펜스(````json`)를 붙이거나 구조를 깨는 문제를 고려해, 서버리스 단에 정규식 기반 전처리, JSON 블록 추출,` JSON.parse()` 다중 Try-Catch 복구 로직을 추가, 이를 통해 포맷 오류가 발생해도 게임 진행이 멈추는 치명적 크래시를 방어.

## 5. 게임 디자인 및 기획 (Game Design)

단순한 기능 구현을 넘어, 유저가 게임에 장기적으로 머물게 하는 게임 디자인 원칙을 적용했습니다.

- **의뢰 시스템 이원화:** 창의성을 발휘하는 `[미적 퀘스트(🎨)]`와 아이디어로 상황을 돌파하는 `[문제 해결 퀘스트(💡)]`를 배치하여 반복 플레이의 지루함을 방지했습니다.
- **유동적 명성(Fame) 시스템:** 결과에 따라 명성(평판) 수치가 오르내리며, 이에 따라 골드 획득량에 버프/디버프가 적용되도록 하여 게임 내 경제 밸런스와 긴장감을 부여했습니다.
- **내러티브(Storytelling):** 각기 다른 사연을 가진 NPC들과 교감하고, 숨겨진 진실을 파헤치는 스토리를 통해 힐링과 성취감을 동시에 제공했습니다.

### 5-1. 모바일 UI 대응 문제

첫 모바일 게임 개발에서 예상보다 크게 부딪힌 문제는 AI보다 **해상도 대응과 UI 레이아웃**이었습니다.

기기마다 화면비, 안전 영역, devicePixelRatio가 달라 캔버스, NPC 대사창, 하단 UI가 충돌하는 문제가 발생했습니다.

이를 해결하기 위해 다음과 같은 방식으로 개선했습니다.

- devicePixelRatio를 반영해 캔버스 내부 버퍼와 표시 크기를 분리
- resize 이벤트 기반으로 캔버스 크기를 동적으로 재계산
- NPC 및 패널 UI를 viewport 기준 위치 계산으로 재배치

다만 이 부분은 아직도 완전히 만족스럽진 않습니다. 여러 해상도를 지원하도록 많이 개선했지만, **UI 유연성이 100% 완성된 상태는 아니며**, 특정 비율의 기기에서는 간격이나 배치가 다소 타이트하게 느껴질 수 있습니다. 

오히려 이 경험을 통해, 모바일 게임에서는 AI 기능 못지않게 **반응형 UI와 해상도 설계 자체가 핵심 엔지니어링 과제**라는 사실을 크게 배웠습니다.

## 6. 비즈니스 임팩트 (Business Impact)

이 프로젝트는 "만들었다"에서 끝나지 않았습니다. 실제로 배포했고, 유저가 들어왔고, 수익이 발생했고, 행정적인 심의 절차까지 모두 완수했습니다.

### 6-1. 유입과 사용자 지표 검증

토스앱 버전 대시보드 기준, **총 `1,291명` 규모의 사용자 데이터**가 확인되었습니다. 주간 단위 WAU 그래프에서는 초기 피크가 약 `400` 수준까지 형성되었고, 이후에도 반복 유입이 이어졌습니다.

- **OS 비중:** Android `65%` / iOS `35%`
- **성별 비중:** 여성 `76.5%` / 남성 `23.5%`
- **인사이트:** '감성형 드로잉/힐링'이라는 장르적 기획이 실제로 타겟 사용자군(20~40대 여성)에게 정확히 반응하고 있음을 지표로 검증했습니다.

## 7. 회고 (Retrospective)

처음에는 저도 *"NPC의 성격과 감정까지 전부 LLM이 만들어주면 더 자연스럽지 않을까?"* 라고 생각했습니다. 
하지만 실제로 부딪혀 보니, 라이브 서비스에서 중요한 건 더 화려한 문장이 아니었습니다.

중요한 건 세 가지였습니다.

1. **같은 입력에 대해 예측 가능한 결과를 줄 것**
2. **비용 구조가 수익 구조를 압도하지 않을 것**
3. **AI가 제품의 규칙을 망치지 않을 것**

그래서 저는 LLM을 줄였습니다. 더 많이 맡기지 않고, 더 적게 맡겼습니다. 대신 그 적게 맡긴 역할을 아주 명확하게 정의했습니다. 그 결과 `꿈의 그림 공방`은 "AI가 들어간 게임"이 아니라, "AI를 통제 가능한 부품으로 설계한 게임"이 되었습니다.

또 하나의 교훈은 모바일 게임에서 AI보다 먼저 해결해야 할 문제가
**해상도 대응과 UI 안정성**이라는 점이었습니다.
첫 모바일 게임이다 보니 다양한 화면비와 DPI, 뷰포트 변화에 유연하게 대응하는 일이 생각보다 훨씬 어려웠고, 실제로 많은 시간을 여기에 썼습니다.
여러 해상도를 지원하도록 개선했지만 아직 완벽하다고 말하긴 어렵습니다.
다만, 실제 서비스에서 부딪힌 문제를 직접 수정하며 제품의 완성도를 높여가는 과정 자체가 엔지니어로서 더 중요한 경험이었다고 생각합니다.

이 프로젝트를 통해 제가 얻은 가장 중요한 철학은 이것입니다.

> **"진짜 엔지니어링은 최고의 기술을 가장 많이 쓰는 것이 아니라, 목적과 비용, 사용자 경험에 맞게 기술의 역할을 정확히 제한하는 데서 시작된다."**
